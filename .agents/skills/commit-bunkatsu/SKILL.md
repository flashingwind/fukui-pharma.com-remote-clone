# Commit Bunkatsu

## 何をするスキルか
作業ツリーの変更を確認し、機能・修正・設定・ドキュメントなどの意味単位に分割してコミットします。
「まず壊さない」「履歴を読みやすくする」を優先します。

## 使うタイミング
- 変更が複数テーマにまたがっている
- 一気にコミットするとレビューしにくい
- 後でrevert/cherry-pickしやすい履歴にしたい
- 依頼例: 「変更を適当にぶんかつしてコミットして」

## 手順
1. 変更の棚卸し
- `git status --short` と `git diff --name-only` で全体を把握する。
- 変更を「目的ごと」に分類する（例: content, redirects, refactor, docs）。

2. 分割方針を決める
- 依存関係の強い変更は同じコミットにまとめる。
- 独立して戻せる変更は分離する。
- 生成物や調査メモは `chore` として分離する。

3. ステージング
- 基本はファイル単位で `git add <path>`。
- 同一ファイルに複数意図が混在する場合は `git add -p` でハンク分割する。
- 既存のユーザー変更を壊さない。不要な巻き戻しはしない。

4. コミット作成
- 各まとまりごとに `git commit -m "<type>(<scope>): <summary>"` を作る。
- 推奨 `type`: `feat`, `fix`, `chore`, `docs`, `refactor`。
- 1コミット1責務を守る。

5. 検証
- `git log --oneline -n <作成数>` で分割結果を確認する。
- `git status` が意図どおり（残件あり/なし）か確認する。
- 必要ならコミット順を調整する（rebaseは明示依頼時のみ）。

## 分岐ルール
- 変更が少数で単一目的: 1コミットで完了。
- 変更が多目的: 目的別に2〜5コミットへ分割。
- 同一ファイルに混在: ハンク分割を優先。難しい場合はユーザーへ確認。
- 不明な生成ファイルがある: 先に由来を確認し、不要なら除外。

## 完了条件
- 各コミットメッセージが内容を説明している。
- 各コミットが独立して理解できる。
- 無関係な変更が混ざっていない。
- 依頼された変更がすべてコミット済みである。

## 出力フォーマット
- 作成したコミットを番号付きで提示する。
- 各コミットに `hash`, `message`, `files` を添える。
- 必要なら次の安全な一手（push前チェック）を1つだけ提案する。
